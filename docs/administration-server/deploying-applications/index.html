アプリケーションのデプロイ

WebLogic Remote Consoleを使用して、WebLogic Serverへのアプリケーションのデプロイメント・プロセスを管理できます。

アプリケーション・デプロイメント・プロセスの一般情報は、「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「WebLogic Serverデプロイメントの理解」を参照してください。

サポートされるデプロイメント・ユニット

デプロイメント・ユニットは、Java EE仕様に従って構成されており、WebLogic Serverにデプロイ可能なJava EEアプリケーション(エンタープライズ・アプリケーションまたはWebアプリケーション)、あるいはスタンドアロンJava EEモジュール(EJB、リソース・アダプタなど)を参照します。

次のデプロイメント・ユニットがサポートされています:

  • エンタープライズ・アプリケーション

  • Webアプリケーション

  • Enterprise JavaBean

  • Webサービス

  • Java EEライブラリ

  • オプション・パッケージ

  • JDBC、JMS、およびWLDFモジュール

  • クライアント・アプリケーション・アーカイブ

各デプロイメント・ユニットの詳細は、「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「サポートされているデプロイメント・ユニット」を参照してください

アプリケーションのインストール

アプリケーションは、アーカイブされたEARファイルまたは展開されたディレクトリとしてインストールできます。 アプリケーションをインストールすると、その物理ファイルまたはディレクトリがWebLogic Serverに認識されます。

このプロシージャは、Java EEライブラリを除いて、「サポートされているデプロイメント・ユニット」にリストされているすべてのデプロイメント・ユニットに適用されます。 Java EEライブラリについては、かわりに「Java EEライブラリのインストール」を参照してください。

  1. 「編集ツリー」で、「デプロイメント」、「アプリケーション・デプロイメント」の順に移動します。

  2. 「新規」を選択します。

  3. アプリケーションの名前を入力します。

  4. アプリケーションのデプロイ先となるサーバーおよびクラスタを選択します。

  5. アーカイブ・ファイルまたは展開されたディレクトリを管理サーバーに認識させます。

    • アプリケーションがファイル・システム上にあり、管理サーバーへのアップロードが必要な場合は、「アップロード」オプションを有効にします。 次に、Sourceの横にある「ファイルの選択」をクリックして、システム上のアプリケーションのロケーションを参照します。
    • アプリケーションがすでに管理サーバーのファイル・システムにある場合は、「アップロード」オプションを無効にします。 次に、「ソース・パス」フィールドに、アプリケーションへのファイル・パスを入力します。

  6. デプロイメント・プランの追加、別のステージング・モードの選択、またはデプロイメント時のアプリケーションの動作の設定を行います。

  7. 「作成」をクリックします。

モニタリング・ツリーで、実行中のデプロイメント・タスクのステータスを表示できます: デプロイメント: デプロイメント・タスク page.

新しいアプリケーションが「アプリケーション・デプロイメント」ノードの下に表示されます。 このページでは、アプリケーションに追加の変更を行うことができます。

クライアント・リクエストを処理する前に、アプリケーションを起動する必要があります。

申請の開始

WebLogic Serverクライアントでアプリケーションを使用可能にするには、アプリケーションを起動する必要があります。

アプリケーションを起動すると、すぐにクライアントで使用できるようにしたり、最初に管理モードで起動して期待どおりに動作するようにできます。 管理モードで起動すると、分散アプリケーションの最終的なサニティ・チェックを、クライアントを中断することなく本番環境で直接実行できます。

  1. モニタリング・ツリーで、「デプロイメント」、「アプリケーション管理」の順に移動します。

  2. 起動するアプリケーションを選択します。

  3. 「開始」をクリックし、次のいずれかを選択します:

    • すべてのリクエストの処理: アプリケーションをすべてのクライアントがすぐに利用できるようにするため。
    • 管理リクエストのみを処理しています: アプリケーションを管理モードでのみ使用できるようにします。

アプリケーションの停止

アプリケーションを停止すると、クライアントがアプリケーションにアクセスできなくなります。 クライアントが使用できないか、管理タスクのみを実行できるように管理モードに移行するかを選択できます。

アプリケーションを停止しても、そのソース・ファイルはサーバーからは削除されません。停止したアプリケーションを再デプロイして、WebLogic Serverクライアントで再度使用可能にできます。

  1. モニタリング・ツリーで、「デプロイメント」、「アプリケーション管理」の順に移動します。

  2. 停止するアプリケーションを選択します。

  3. 「停止」をクリックし、次のいずれかを選択します:

    • 作業完了時: アプリケーションが作業を完了し、現在接続されているすべてのユーザーが切断できるようにするため。
    • 今すぐ強制停止: 実行中の作業や接続しているユーザーに関係なく、アプリケーションをただちに停止します。
    • 非管理リクエストの処理: すべての作業が終了したらアプリケーションを停止し、管理目的でアプリケーションにアクセスできるように管理モードにします。

デプロイメント・プロパティの指定

アプリケーションのデプロイ後に、環境のニーズにあわせて追加の設定を構成できます。

  1. まだ作成していない場合は、アプリケーションのデプロイメント・プランを作成します。 デフォルトの構成設定は、プランを作成せずに表示できます。ただし、デプロイメント・プランを作成するまでは読取り専用です。 詳細については、「デプロイメント計画」を参照してください。

    ノート

    自動デプロイされたアプリケーションのデプロイメント・プランは作成できません。

    1. 「モニタリング・ツリー」で、「デプロイメント」、「アプリケーション管理」、myAppの順に移動します。

    2. 「プランの作成」をクリックします。

    3. 「計画パス」フィールドに、新しいデプロイメント・プランのファイル・パスを入力します。 デプロイメント・プランはXML形式である必要があり、plan.xmlと呼ばれます。

      可能な場合は、アプリケーションのルート・ディレクトリのplan/サブディレクトリに、単一のアプリケーションの新しいデプロイメント・プランを作成する必要があります。

    4. 「完了」をクリックします。 WebLogic Serverは、基本的なデプロイメント・プランを作成します。

      アプリケーションのデプロイメント・プランは、デプロイメントで確認できます: アプリケーション管理: myApplication: デプロイメント・プラン(拡張).

  2. 「デプロイメント」、「アプリケーション管理」、myApplication、「構成」の順に移動します。 構成ノードとその子を調べ、使用可能なデプロイメント構成オプションを確認します。

    ノート

    構成ノードとその子の内容は、アプリケーション・タイプによって異なります。 たとえば、webアプリケーションにはコンテナ記述子の設定が含まれ、リソース・アダプタにはアウトバウンド接続プールの設定が含まれます。

  3. 変更する場合は、「保存」をクリックします。 構成ノードとその子に加えた変更は、アプリケーションのデプロイメント・プランに自動的に反映されます。

  4. アプリケーションを更新し、場合によっては再デプロイして変更を適用します。 手順については、「アプリケーションの更新または再デプロイ」を参照してください。

デプロイメント・プラン

デプロイメント・プランを使用して、アプリケーションのデプロイメント・プロパティ値を指定します。

デプロイメント・プランは、アプリケーションのデプロイメント記述子を操作またはオーバーライドして、特定のWebLogic Server環境へのデプロイメント用にアプリケーションを構成するオプションのドキュメントです。 デプロイメント・プランはXMLで記述されます。

「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「WebLogic Serverデプロイメント・プランの理解」を参照してください。

ノート

自動デプロイされたアプリケーションのデプロイメント・プランは作成できません。

デプロイメント・プランの説明と例については、「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「デプロイメント・プランの内容の理解」を参照してください。

通常、アプリケーションのデプロイメント・プロパティを変更する必要がある場合は、デプロイメント・プランを手動で編集するかわりに、構成ノード(「デプロイメント・プロパティの指定」を参照)を使用する必要があります。

デプロイメント・プランの変更に非動的変更が含まれる場合は、アプリケーションを再デプロイして、デプロイメント・プランからアプリケーションに変更を伝播する必要があります。

デプロイメント・プランの変更

デプロイメント・プランは、アプリケーションの新しいデプロイメント手順で手動で更新できます。

詳細は、「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「デプロイメント・プランの手動カスタマイズ」を参照してください。

  1. 「モニタリング・ツリー」で、「デプロイメント」、「アプリケーション管理」、myApplication、「デプロイメント・プラン(拡張)」の順に移動します。

  2. デプロイメント・プラン全体またはデプロイメント・プラン内の個々の変数割当てを編集できます:

    • デプロイメント計画ドキュメントを編集するには:
      1. 「デプロイメント・プラン」タブを選択します。
      2. テキスト・ボックス・フィールド内でデプロイメント・プランを直接編集します。 また、プランの内容を外部テキスト・エディタにコピーし、編集を行ってから、変更したデータをWebLogic Remote Consoleに貼り付けることもできます。
      3. 「保存」をクリックします。
    • 個々の変数割当てを編集するには:
      1. 「変数割当」タブを選択します。
      2. 編集する変数割当てモジュールを選択し、「編集」をクリックします。
      3. 値を入力し、配列を更新する場合は操作を選択します。
      4. 「完了」をクリックします。

  3. アプリケーションを更新し、場合によっては再デプロイして変更を適用します。 手順については、「アプリケーションの更新または再デプロイ」を参照してください。

アプリケーションの更新または再デプロイ

デプロイされたアプリケーションまたはそのデプロイメント・プランに変更を加えた後は、新しい手順でアプリケーションを更新し、場合によってはアプリケーションを再デプロイして、変更をWebLogic Serverクライアントで使用できるようにする必要があります。

ノート

変更が動的な場合は、デプロイメント・プランの更新のみが必要です。 ただし、変更に非動的変更が含まれる場合は、アプリケーションを再デプロイして、デプロイメント・プランからアプリケーションに変更を伝播する必要があります。

デプロイメント・プランを更新しようとしたが、変更に再デプロイメントが必要な場合は、WebLogic Remote Consoleによって、かわりにアプリケーションを再デプロイするよう求められます。

アプリケーションが本番環境にデプロイされている場合は、「Oracle WebLogic Serverへのアプリケーションのデプロイ」の「再デプロイメント戦略の概要」で、アプリケーションの停止時間の制限に関するガイダンスを確認します。

  1. 「モニタリング・ツリー」で、「デプロイメント」、「アプリケーション管理」の順に移動し、更新するアプリケーションを選択します。

  2. Update/Reployをクリックし、次のいずれかのオプションを選択します:

    • 更新 - サーバー上のデプロイメント・プラン: サーバー上にある新しいデプロイメント・プランを使用してアプリケーションを更新します。 このオプションは、すべての変更が動的である場合に使用します。
    • 更新 - ローカル・マシン上のデプロイメント・プラン: ローカル・マシンにある新しいデプロイメント・プランを使用してアプリケーションを更新し、管理サーバーのアップロード・ディレクトリにアップロードします。 このオプションは、すべての変更が動的である場合に使用します。
    • 再デプロイ - サーバー上のデプロイメント・ソースおよびプラン: サーバーにあるデプロイメント・プランを更新し、アプリケーションを再デプロイします。 アプリケーションを再起動する必要がある動的でない変更が変更に含まれている場合は、このオプションを使用します。
    • 再デプロイ - ローカル・マシン上のデプロイメント・ソースおよびプラン: ローカル・マシンにあるデプロイメント・プランを更新し、アプリケーションを再デプロイします。 アプリケーションを再起動する動的でない変更が変更に含まれている場合は、このオプションを使用します。

  3. 必要に応じて、新しいデプロイメント・プランへのファイル・パスを入力します。

  4. 「完了しました」をクリックして、アプリケーションを更新または再デプロイします。

アプリケーション・デプロイメントのステータスを追跡できます。 まだ「モニタリング・ツリー」にある状態で、「デプロイメント」、「デプロイメント・タスク」に進みます。

アプリケーション・デプロイメントのテスト

アプリケーションのデプロイ後に、正常にデプロイされたことを確認できます。

  1. 「モニタリング・ツリー」で、「環境」、「サーバー」、myServer、「デプロイメント」、「アプリケーション・ランタイム」、myApp、「コンポーネント・ランタイム」、myServer/myApp、「サーブレット」の順に移動します。

  2. ターゲット・サーバー・アドレスと、サーブレット表のContextPathおよびName値を使用してURLを作成します。 たとえば、管理サーバーにデプロイされたアプリケーションの場合、テストURLは次のようになります: http://adminserver:7001/myapp/welcome。

    ノート

    アプリケーション・デプロイメントに影響を与える可能性のあるネットワーク・チャネルに加えた変更を考慮してください。 たとえば、アプリケーションが別のポート上にある場合があります。

  3. ブラウザにURLを入力します。 ページが正しくロードされると、アプリケーションは正常にデプロイされました。

ドメインからのアプリケーションの削除

インストールされているアプリケーションまたはモジュールは、明示的に削除するまで、停止、再起動または更新するためにドメイン内で引き続き使用できます。 ドメインからアプリケーションまたはモジュールを削除した後、アプリケーションまたはモジュールを再デプロイして、WebLogic Serverクライアントで再度使用できるようにする必要があります。

デプロイされているアプリケーションをWebLogic Serverクライアントで一時的に使用できなくするには、そのアプリケーションを削除するのではなく停止することを検討してください。 「アプリケーションの停止」を参照してください。

  1. 「編集ツリー」で、「デプロイメント」、「アプリケーション・デプロイメント」の順に移動します。

  2. 削除するアプリケーションを選択し、「削除」をクリックします。

Java EEライブラリのインストール

Java EEライブラリは、スタンドアロンのEJBまたはWebアプリケーション・モジュール、エンタープライズ・アプリケーション(EAR)にパッケージ化された複数のEJBまたはWebアプリケーション・モジュール、またはデプロイメント時にJava EEアプリケーション・コンテナに登録された単一のプレーンJARファイルです。 ライブラリがインストールされ起動されると、他のデプロイ済みのモジュールが、ライブラリを参照できるようになります。

Java EEライブラリをインストールすると、その物理ファイルまたはディレクトリがWebLogic Serverに認識されます。 Java EEライブラリは、アーカイブされたEARファイルまたは展開されたディレクトリとしてインストールできます。 Java EEライブラリをインストールした後、他のデプロイ済モジュールでそのライブラリの使用を開始できます。

  1. 「編集ツリー」で、「デプロイメント」、「ライブラリ」の順に移動します。

  2. 「新規」をクリックします。

  3. Java EEライブラリの名前を入力します。

  4. Java EEライブラリをデプロイするサーバーおよびクラスタを選択します。 Java EEライブラリを参照するモジュールまたはアプリケーションがデプロイされるすべてのサーバーおよびクラスタをターゲットにしてください。

  5. インストールするJava EEライブラリ(アーカイブ・ファイルまたは展開されたディレクトリ)を管理サーバーに認識させます。

    • Java EEライブラリがファイル・システム上にあり、管理サーバーへのアップロードが必要な場合、「アップロード」を有効にします。 次に、「ファイルの選択」をクリックして、システム上のライブラリのロケーションを参照します。
    • Java EEライブラリがすでに管理サーバーのファイル・システムにある場合は、「アップロード」オプションを無効化し、Java EEライブラリへのファイル・パスを入力します。 展開されたディレクトリを指定すると、WebLogic Serverでは、指定されたディレクトリ以下の階層にあるすべてのコンポーネントがインストールされます。

    ノート

    次のタイプのアーカイブ・ファイル(または対応する展開ディレクトリ)のみをJava EEライブラリとしてインストールできます: EJB JAR、WebアプリケーションWAR、EJB JARまたはWARを含むEARファイル、またはコンパイルされたクラスを含むプレーンJARファイル。

  6. 別のステージング・モードを選択します。

  7. 「作成」をクリックします。

新しいライブラリが「Libraries」ノードの下に表示されます。 このページでは、Javaライブラリに追加の変更を加えることができます。

Java EEライブラリの再デプロイ

Java EEライブラリを更新すると、WebLogic Serverによってアーカイブ・ファイルまたは展開されたディレクトリが再デプロイされます。 ライブラリに変更を加えており、その変更内容を、ライブラリを使用しているWebLogic Serverにデプロイ済みのモジュールおよびアプリケーションで使用できるようにする場合に、ライブラリの更新を行います。

  1. 「モニタリング・ツリー」で、「デプロイメント」、「ライブラリ管理」の順に移動します。

  2. 更新するJava EEライブラリを選択します。

  3. ライブラリの更新方法を決定します:

    • ライブラリへの変更が管理サーバーですでに使用可能であり、ライブラリを使用しているモジュールおよびアプリケーションでのみそれらの変更を使用可能にする場合は、「再デプロイ」を選択します。

    • 新しいアーカイブ・ファイルまたは展開されたディレクトリをアップロードする場合は、「アップロードおよび再デプロイ」を選択します。

      Sourceフィールドに、新しいアーカイブ・ファイルまたは展開されたディレクトリへのパスを入力します。 ライブラリの最新のアーカイブが、管理サーバーのuploadディレクトリにあることを確認します。

  4. 「完了」をクリックします。

WebアプリケーションのJASPICの構成

JASPICを使用している場合は、特定のWebアプリケーションに適用する認証構成プロバイダを指定できます。

Java Authentication Service Provider Interface for Containers (JASPIC)仕様は、サービス・プロバイダ・インタフェース(SPI)を定義します。 JASPIC SPIは、サーバーのWebアプリケーション・メッセージ処理に統合できるメッセージ認証メカニズムを実装する認証プロバイダによって使用されます。

JASPIC仕様は、「JSR 196: コンテナ用のJava認証サービス・プロバイダ・インタフェース」で確認できます。

  1. まだ構成していない場合は、ドメインでJASPICを構成します。

    1. ドメインでJASPICが有効になっていることを確認します。 「編集ツリー」で、「環境」、「ドメイン」の順に移動します。 「セキュリティ」タブで、「拡張フィールドの表示」をクリックし、「JASPIC有効」がオンになっていることを確認します。

    2. 認証構成プロバイダを構成します。

      • WebLogic Server認証構成プロバイダを構成するには、「Oracle WebLogic Serverのセキュリティの管理」の「WLS認証構成プロバイダの作成」を参照してください。
      • カスタム認証構成プロバイダを構成するには、「Oracle WebLogic Serverのセキュリティの管理」の「カスタム認証構成プロバイダの作成」を参照してください。

  2. 「モニタリング・ツリー」で、「デプロイメント」、「アプリケーション管理」、myWebAppの順に移動します。

  3. アプリケーションにデプロイメント・プランがない場合は、「プランの作成」をクリックします。

    WebLogic Serverは、新しく作成した「構成」および「デプロイメント・プラン(拡張)」子ノードを使用して編集できるアプリケーションの基本的なデプロイメント・プランを作成します。

    可能な場合は常に、デプロイメント・プランを直接編集するのではなく、「構成」ノードを使用してデプロイメント・プロパティを編集します。 「デプロイメント・プロパティの指定」を参照してください。

  4. 「構成」ノードを展開し、「JASPICコンテナ」ノードに移動します。

    「JASPICコンテナ」ノードは、webアプリケーションでのみ使用できます。

  5. 構成し、このwebアプリケーションに適用する認証構成プロバイダの名前を入力します。

  6. 「保存」をクリックします。

  7. webアプリケーションを再デプロイするか、サーバーを再起動します。

アプリケーション・スコープ資格証明マッピングの作成

資格証明マッピングを使用して、WebLogicリソースとリモート・システム間のアクセスを制御します。 WebLogic Serverを使用すると、アプリケーションごとにアクセスを制限できます。

  1. WebLogicリソースとそのセキュリティ・データ間の接続を形成するために必要なMBean構成データを決定します。

    1. 「セキュリティ・データ・ツリー」で、「レルム」、myRealm、「資格証明マッパー」の順に移動します。

    2. 資格証明マッピング用に準備するWebLogicリソースに到達するまで続行します。 ノード・パスの例については、「資格証明マッピングのリソースを特定」を参照してください。

    3. 「新規」をクリックし、構成データを取得する必要があるすべての必須属性(アスタリスクでマークされたフィールド)をノートにとります。

    4. 「取消」をクリックします。

  2. 「モニタリング・ツリー」で、「デプロイメント」、「アプリケーション・ランタイム・データ」、myApplication、「コンポーネント・ランタイム」、myApplicationの順に移動します。

  3. 「コンポーネント・ランタイム」表で、アプリケーションの行を見つけますが、クリックしないでください。 かわりに、「EISリソースID」列のアプリケーションのセルを右クリックし、「セルをクリップボードにコピー」を選択します。

  4. 後のステップで使用するために、値をテキスト・ファイルに貼り付けます。

  5. 資格証明マッピングのリソースを識別するために必要な追加データ(ステップ1で決定)を見つけます。 必要なデータは、リソース・タイプによって異なります。

    例えば:

    • EJBにはEJB名が必要です。 「デプロイメント」ノードで、「アプリケーション・ランタイム・データ」、myApplication、「コンポーネント・ランタイム」、myApplication、「EJBランタイム」の順に移動します。
    • リソース・アダプタには、アウトバウンド接続プール・インスタンス名が必要です。 「デプロイメント」ノードの下で、「アプリケーション管理」、myApplication、「構成」、「アウトバウンド接続プール・グループ」、myOutboundConnectionPoolGroup、「アウトバウンド接続プール・インスタンス」の順に移動します。

  6. 「資格証明マッピングのリソースを特定」の説明に従って、リソースとそのセキュリティ・データ間の接続を形成します。 フィールドに入力するには、モジュール、名前、および前のステップで取得したその他のリクエストされた値を使用します。

    このプロセスでは、アプリケーションの最初の資格証明マッピングも作成されます。

  7. さらに資格証明マッピングを作成する場合は、「資格証明マッピングの追加」を参照してください。
